# BYGLab

Engine modeling suite. Core solver: a 1-D finite-volume gas-dynamics + 0-D cylinder model, written in Rust, compiled to WebAssembly, and run **entirely client-side in the browser** — no server-side compute. BMW S54B32 is the reference engine.

## Architecture

| Layer | Stack |
|-------|--------|
| Solver | `rust/crates/byglab-core` — pure Rust, no wasm-specific deps, `cargo test`-able natively |
| Browser binding | `rust/crates/byglab-wasm` — thin `wasm-bindgen` wrapper, no physics of its own |
| Native CLI | `rust/crates/byglab-cli` — same core, JSON in/out, for fast local iteration and reference-data comparison |
| Web | React + Vite + Recharts (`services/web`), solver runs in a Web Worker via wasm |
| Local run | Docker Compose (`web` on :8080) — static SPA, single container, no backend |
| Solver validation | `benchmarks/openwam/` — legacy OpenWAM reference cases (exact Riemann solutions, physically-validated single- and 6-cylinder S54B32 engine models) used as ground truth for the Rust port |

Future modules (Flowbench, Dyno, Turbo, Camshaft, Exhaust, Intake, Combustion) appear in the nav as Coming Soon.

## Quick start (Docker)

```bash
docker compose up --build
```

- App: http://localhost:8080

## Local development (without Docker)

### Rust solver

Requires `rustup` (native install, not Docker — Rust cross-compiles to `wasm32-unknown-unknown` fine on any host) and `wasm-pack`:

```bash
rustup target add wasm32-unknown-unknown
cargo install wasm-pack
```

Fast core-logic iteration (no wasm/browser involved — this is where nearly all solver work happens):

```bash
cd rust
cargo test
```

Rebuild the wasm package after touching `byglab-wasm`'s binding surface:

```bash
wasm-pack build rust/crates/byglab-wasm --target web --dev   # fast, for iteration
wasm-pack build rust/crates/byglab-wasm --target web         # release, for production builds
```

### Web

```bash
cd services/web
npm install
npm run dev
```

The Worker (`src/wasm-worker.js`) imports the wasm-pack output via a relative path into `rust/crates/byglab-wasm/pkg/` — rebuild wasm (above) after Rust changes; Vite picks up the new `pkg/` files on refresh.

### CLI (fast native iteration / reference comparison)

```bash
cd rust
cargo run -p byglab-cli -- crates/byglab-cli/example-config.json
```

## Solver porting roadmap

**Phase 1 — done.** `byglab-core` is a real quasi-1D finite-volume gas-dynamics solver: second-order MUSCL-Hancock scheme (piecewise-linear reconstruction with a minmod slope limiter, HLLC approximate Riemann solver, explicit CFL-limited time-stepping — see `reconstruction.rs`), supporting networks of pipes joined by same-area junctions with fully symmetric second-order accuracy at the junction faces too, variable cross-sectional area (taper — velocity stacks, diffusers, header collector cones), and optional wall friction/heat transfer (`source_terms.rs`). A clean-room Rust reimplementation informed by OpenWAM's validated physics, not a line-by-line C++ translation. Started as a simpler first-order, constant-area, frictionless/adiabatic scheme, then upgraded twice once each stage was validated — all stages checked against real ground truth:

- `tests/quiescent.rs` — uniform at-rest closed pipe stays at rest (no discretization error possible to hide behind).
- `tests/acoustic_resonance.rs` — closed-closed resonance period matches the exact `T = 2L/c0` to **0.09%** (OpenWAM's own higher-order scheme: 0.07% — comparable; a first-order-only version of this solver measured 0.002% here, tighter still, since numerical dissipation for a small linear disturbance mostly damps amplitude rather than shifting phase — the slope limiter changes that balance slightly, still well within margin either way).
- `tests/sod_shock_tube.rs` — compared pointwise against a from-scratch Rust port of Toro's exact Riemann solver (`tests/support/exact_riemann.rs`, self-tested against the textbook Sod case). Star region matches to 4 significant figures (32 Pa / 0.07 m/s); whole-profile RMS error (0.075 bar pressure, 8.2 m/s velocity) is concentrated at the shock/rarefaction fronts and now runs at essential parity with OpenWAM's own TVD-scheme result (0.068 bar / 6.8 m/s) — a first-order-only version of this solver measured roughly 2.5x worse on both (0.186 bar / 14.3 m/s), confirming the MUSCL-Hancock upgrade's benefit directly on this case. Whole-profile RMS in the low tens of Pa isn't a reachable target for any practical shock-capturing scheme — a genuine discontinuity is always smeared over a few cells no matter the scheme order, and refining the mesh only helps at roughly `sqrt(cell count)`; the *smooth* star-region result above (32 Pa) is where sub-100 Pa accuracy is actually achievable.
- `tests/mesh_convergence.rs` — doubling mesh resolution on the Sod case shrinks error at close to the scheme's design rate: smooth star-region error shrinks at order 3.9 (pressure) / 2.6 (velocity) — even *exceeding* the formal 2nd-order rate, since that probe's only error source is decaying numerical diffusion bleeding in from the distant shock, not classical truncation error — while whole-profile RMS error (which includes the shock/rarefaction fronts) shrinks at a reduced but real order 0.55 / 0.58, the expected signature of discontinuity smearing capping the rate below the smooth-region behavior. A meaningfully stronger correctness signal than a fixed-tolerance check alone — a subtly broken scheme could still look "small enough" at one resolution while failing to converge under refinement.
- `tests/tapered_pipe_stays_at_rest.rs` — the well-balanced check for the taper geometric source term: a uniform, at-rest gas column in a pipe tapering from ⌀0.05m to ⌀0.10m generates **zero** spurious velocity (< 1e-6 m/s), proven algebraically exact (not just empirically small) by construction — the source term reuses the same MUSCL-reconstructed face pressures already computed for the flux, rather than a separately-averaged value, so the two cancel exactly for any uniform state regardless of how the area varies.
- `tests/isentropic_nozzle.rs` — steady subsonic flow through a converging duct, calibrated from one interior station's own state and checked against every other interior station via the exact isentropic area-Mach relation (`tests/support/isentropic_nozzle.rs`, a from-scratch bisection solver, self-tested against a textbook A/A* table value). Interior pressure matches to **0.11%** of the driving pressure range, comfortably inside the 0.2% target — validates that the taper source term correctly reproduces quasi-1D isentropic flow acceleration under genuine (non-zero-velocity) flow, isolated from the separately-documented approximation in the `Reservoir` boundary condition (see `boundary.rs`).
- `tests/shu_osher.rs` — the classic Shu-Osher shock/entropy-wave interaction problem (a Mach-3 shock advancing into a sinusoidal density field, `gamma=1.4`, standard non-dimensional problem values, run to `t=1.8`). Checks that pressure stays at its initial uniform value in the region the shock has not yet reached — measured deviation was **exactly `0.0`**, bit-for-bit, not just "small." A stronger property than it first looks: since velocity and pressure are both spatially uniform ahead of the shock while density varies sinusoidally, and this solver's MUSCL reconstruction limits each primitive variable independently (no cross-variable coupling), every face flux ahead of the shock is *exactly* constant regardless of the density gradient, so nothing should change there at all until the shock's actual domain of dependence reaches it — a real correctness signature, not just "small enough," and the measured result confirms the scheme has no hidden cross-variable coupling.
- `tests/heat_transfer_relaxation.rs` — a quiescent gas column with a fixed differing wall temperature relaxes toward it exactly along the closed-form exponential time constant `τ = ρ·cv·D/(4h)`, matching to **0.0003 K** out of a ~107 K total temperature rise. Also regression coverage for a zero-Reynolds-number singularity caught during design review (naive friction/heat-transfer formulas are singular at u=0 — routine in pulsating engine flow — which would otherwise produce `NaN` instead of the physically correct zero).
- `source_terms.rs`'s own unit tests check the Haaland friction factor and Colburn Nusselt-number correlations against Moody-chart/textbook reference points.
- Mass conservation in a closed pipe with a moving pressure pulse (mass has no source term — flux differencing conserves it exactly regardless of scheme details, so this is a bookkeeping check, not a physics one): **0.001%** for a constant-diameter pipe (`solver.rs`'s own unit test) and **0.000145%** for a tapered pipe (`tests/mass_conservation_taper.rs`, checking the area-weighted flux differencing specifically — a genuinely different code path added for taper support), both far inside the 0.1% target.

Wall friction (Haaland correlation) and heat transfer (Colburn correlation, fixed wall temperature) are simpler standard choices deliberately *not* ported from OpenWAM's own piecewise-polynomial Colebrook cascade, regime-switched Nusselt correlations, and full multi-layer transient wall-conduction model (`benchmarks/openwam/OpenWAM/Source/1DPipes/TTubo.cpp`) — same governing physics, independently re-derived and cross-checked against OpenWAM's formulas, simpler code. A junction between two mismatched-area pipe ends (a genuine sudden area change, needing a different loss-coefficient-based treatment) is explicitly out of scope and rejected at runtime with a clear panic.

Junctions (phase 2 of the original plan) landed as part of phase 1, since both multi-pipe validation cases needed them. Remaining phases, scoped against the end goal of a full engine-modeling suite (intake manifold, exhaust header/muffler, camshaft, cylinder head porting, valve sizing/seat angle, and throttle body optimization studies):

2. **0-D cylinder + Wiebe combustion + valve flow + camshaft**, checked against the single-cylinder S54B32 OpenWAM case. Cylinder: crank-slider `V(θ)`, single/double Wiebe heat release, Woschni-type wall heat transfer. Valve flow: quasi-steady compressible orifice flow through the curtain area (the same subsonic/choked-flow physics already validated by `tests/isentropic_nozzle.rs`), scaled by a discharge coefficient `Cd(lift/diameter)`. **Scoping decision: `Cd` curves are a parametric *input*, not predicted from 3D port geometry** — this is how professional 1D tools (GT-Power, WAVE) handle it too, and it's what makes "cylinder head porting," "valve seat angle," and "valve size" studies tractable: you compare different `Cd` curves (measured on a flow bench, or estimated), not run port-level CFD inside the cycle solver. Camshaft: a lift-vs-crank-angle profile (lift/duration/timing/ramps or a direct table) per valve, feeding the curtain area; must support reverse flow (blowdown, overlap reversion), since that's the actual physics of exhaust pulse tuning. Throttle body reuses this same orifice-flow submodel with a `Cd(throttle angle)` curve instead.
   - **Crank mechanism — done.** `crank_mechanism.rs`: exact (finite connecting-rod-length, not the infinite-rod/simple-harmonic-motion approximation) slider-crank kinematics — piston position, velocity, and acceleration as functions of crank angle, with piston pin offset ("desaxé") support. Offset shifts true TDC away from the "crank pin collinear with the cylinder axis" reference — a real second-order effect, solved exactly via a one-time Newton iteration at construction (not a small-angle series approximation) so every public method's crank angle is measured from *true* TDC, matching how a physical TDC sensor is calibrated. Validated against the standard textbook slider-crank position/velocity formulas (exact match, no offset case), central finite differences of the solver's own position/velocity (velocity and acceleration respectively), the simple-harmonic-motion limit as rod length → ∞, and an exact identity (piston velocity is exactly zero at true TDC and true BDC, with or without offset — confirmed to < 1e-9 m/s). One genuine finding along the way: with offset, total TDC-to-BDC piston travel is *not* exactly the nominal stroke (TDC and BDC each get an independent, generally unequal phase correction) — measured ~0.003-0.007% deviation for realistic 1-1.5mm offsets, a real physical effect now documented rather than assumed away.
   - **Motoring cylinder model — done.** `cylinder.rs`: cylinder volume `V(θ)` built on `CrankMechanism`, plus a first-law energy balance (`dU = -p dV`) for the trapped charge, integrated in crank-angle space (not real time — a reversible adiabatic process is exactly rate-independent) via classic 4th-order Runge-Kutta. Deliberately scoped to *motoring only* (no combustion, no valve mass flow, no wall heat transfer yet) because that has an exact closed-form answer to validate against before adding physics that don't: the isentropic relation `p·V^γ = const`. Using real S54B32 geometry (CR 11.5:1), compressing from BDC to TDC matches the exact isentropic pressure and temperature to **~1e-11 relative error** — essentially machine precision, not just "small." Mass conservation is exact (no valve flow exists in this model yet, so there's nothing to leak). A full BDC→TDC→BDC round trip — compression then expansion retracing the same reversible path — returns to the exact initial energy and pressure to **~6.6e-14 relative error**, confirming the numerical integration doesn't manufacture or leak energy over a closed cycle. Halving the RK4 step size shrinks the error at **observed order 4.26**, confirming genuine 4th-order convergence rather than a scheme that merely looks accurate at one step count.
   - **Wiebe combustion + Woschni wall heat transfer — done.** `combustion.rs` (mirroring `source_terms.rs`'s pattern) adds two source terms to the same energy balance: `dU/dθ = -p·dV/dθ + dXB/dθ·Q_total + (dQ_wall/dt)/ω` (`Cylinder::integrate_fired_cycle`, alongside — not replacing — `integrate_motoring`). Formulas (Wiebe mass-fraction-burned, Woschni `h`, three-surface wall heat loss) ported directly from OpenWAM's own C++ source, not textbook-only, and cross-checked against the real single-cylinder S54 OpenWAM reference case (`benchmarks/openwam/cases/engine_s54_2500rpm/`) — a design review against that same source caught one real bug before it shipped (the Woschni combustion-turbulence term was initially gated "on forever after ignition"; OpenWAM only activates it during the Wiebe burn window itself). Validated three ways: (1) cylinder volume at IVC matches OpenWAM's own reported value to **0.085%**, confirming the crank-angle convention lines up; (2) RK4 convergence measured *right at* the ignition point comes out at order **1.00**, not 4 — a real, understood effect (the Wiebe shape factor `m=2.5` is non-integer, so `dXB/dθ`'s higher derivatives are singular exactly at ignition, the same "convergence-order-capped-near-a-non-smooth-point" phenomenon already documented for a shock in `mesh_convergence.rs`) — confirmed by re-measuring 10° away from ignition, where order recovers to **5.98**, proving the RK4 implementation itself is correct; (3) seeding the model with OpenWAM's own reported IVC state (extracted directly from its output trace) and running the full combustion trace comes out a **systematic ~30% high on pressure, ~40% high on temperature** — same peak-pressure/peak-temperature timing as OpenWAM (~15°/~22° ATDC respectively, matching exactly), but too much energy released. The likely cause, clearly identified rather than hidden: total heat release is estimated from the *entire* IVC trapped mass at a stoichiometric ratio, with no residual (burned) exhaust gas fraction subtracted out — this model doesn't track residuals yet, so it's over-crediting how much of the trapped charge is fresh fuel+air. A calibration input (or residual-fraction modeling) is the natural next refinement if tighter absolute agreement is wanted; the underlying Wiebe/Woschni physics and RK4 integration are independently confirmed correct by the other two checks. `combustion.rs` also exposes `crank_angle_at_mass_fraction_burned` — the exact closed-form inverse of the Wiebe function (`XB = 1-exp(-C*y^(m+1))` inverts directly, no iteration needed) — for standard combustion-phasing metrics CA10/CA50/CA90. Validated against an independently-coded bisection on the forward function (not just round-tripping the same formula): agrees to ~1e-15 (floating-point precision). Real numbers for the S54 case: **CA10=-1.38°, CA50=8.34°, CA90=17.89° ATDC** — CA50 lands right in the 6-10° ATDC range real gasoline engines target for best-efficiency (MBT) timing, a physical sanity check alongside the mathematical one.
   - **Ideal Otto cycle — done.** A stronger validation target than the OpenWAM comparison above, since it has a genuinely exact answer with no unmodeled-physics uncertainty: adiabatic compression (`integrate_motoring`, already validated) → instantaneous constant-volume heat addition (exact by definition — just adding energy at fixed volume) → adiabatic expansion (`integrate_motoring` again) reproduces the textbook efficiency `η = 1 - 1/CR^(γ-1)` to **4.6e-12 relative error**, with all three intermediate states (T2, T3, T4) matching their exact closed-form values to ~1e-12 as well — confirms the energy-balance bookkeeping itself, independent of the empirical Wiebe/Woschni correlations. IMEP (indicated mean effective pressure, `W_net/V_displaced`) checked two independent ways against the same exact target: the energy-balance shortcut (`Qin-Qout`) matches to **4.7e-15**, and a genuinely separate numerical path — trapezoidal integration of `p dV` along the actual simulated compression/expansion trajectory, not just an endpoint-state shortcut — matches to **1.3e-6** at 2000 segments. That trapezoidal figure was confirmed to be pure, expected discretization error (not a bug) by checking its scaling: 10x more segments gave exactly 100x less error, the textbook O(1/N²) trapezoidal convergence rate.
   - **Camshaft + valve flow + cylinder "breathing" — done, standalone.** `camshaft.rs` (a raised-cosine/versine lift profile) and `valve.rs` (curtain area + quasi-steady compressible orifice mass flow, choked/subsonic, correct reverse-flow handling for valve overlap) are new, plus a third independent `Cylinder` integration mode, `integrate_breathing` (`dm/dθ = ṁ/ω`, `dU/dθ = -p·dV/dθ + h_flow·ṁ/ω`), validated with a cylinder breathing to/from a fixed reservoir. A design review re-derived the compressible-flow formulas independently and caught one real, non-obvious bug before it shipped: the curtain-area formula used `sin(seat_angle)` where it should be `cos(seat_angle)` — invisible at exactly 45° (where they're equal) but a ~40% area error at other seat angles. Debugging the first test run also caught a real robustness gap in `crank_mechanism.rs`: a zero-stroke ("rigid, fixed-volume") mechanism — a legitimate way to model a rigid 0D element by reusing the same kinematics code — divided `0.0/0.0` in the TDC-finding Newton solve (both derivatives are identically zero when nothing moves), poisoning every subsequent query with `NaN`; fixed by returning the initial guess directly in that degenerate case. Validated: (1) a closed valve reduces `integrate_breathing` to exactly `integrate_motoring`'s result; (2) under guaranteed-choked conditions (proven via a closed-form pressure-rise bound, not hoped), mass/energy/pressure all grow **exactly linearly** in a rigid cylinder, matching the closed-form choked-flow prediction to **3.3e-12 relative error**; (3) a realistic camshaft-driven intake-like window (correctly centered on BDC, not TDC — an earlier version centered on TDC and got net *outflow* from piston compression dominating the balance, a real effect but the wrong scenario for testing intake breathing) shows genuine net inflow from the higher-pressure reservoir, with an independent trapezoidal integration of the mass flow rate agreeing with the RK4 result to **5.5e-9**; (4) RK4 convergence checks turned up a genuine surprise: the lift profile's 2nd-derivative jump at the valve open/close events, expected to mildly reduce convergence order the way Wiebe combustion's non-integer exponent does, **measured no reduction at all** (order ~4.0 either way) — but the compressible-flow correlation's choked/subsonic branch switch *did* cause a real, if step-grid-sensitive, degradation (order 0.74 in one specific case) before test conditions were adjusted to stay within a single flow regime.
   - **Real BMW S54 camshaft data — ingested.** `camshaft_presets.rs` holds 10 real Schrick camshaft grinds (5 intake, 5 exhaust — low/medium/high street tiers) ingested directly from official Schrick data sheets, not estimated: exact duration, valve lift, and installed open/close timing for every published spread option, plus the manufacturer's own `@1mm` cross-check figures (used only to check this crate's versine approximation, never to build the profile itself). One real subtlety found while ingesting: Schrick's data sheets reference crank angle "0°" to the *gas-exchange* TDC (between exhaust and intake strokes), not the firing/combustion TDC `combustion.rs`'s Wiebe timing uses — confirmed by checking that the exhaust lobe centerlines (e.g. -104°) land on physically ordinary EVO/EVC figures (72° BBDC / 44° ATDC) only under that interpretation. Combining this data with the combustion model later will need an explicit ±360° shift; not needed for the current standalone breathing simulation, since `Cylinder::volume` is exactly 360°-periodic. Checking the idealized versine shape against the manufacturer's own `@1mm` timing across all 10 grinds shows a consistent, honest **7-12° discrepancy** (the real cam lobes ramp up faster than a symmetric raised-cosine) — a real, quantified limitation of the versine approximation, not hidden, and exactly the reason a direct-lift-table `CamProfile` variant would be needed for high-fidelity valve-event timing later.
   - **Real 1D pipe-network coupling for cylinder breathing — done.** `valve_port.rs` binds a valve/cam pair (`ValvePort`) to one end of a real `pipe::Pipe`, replacing the fixed-reservoir state above with the pipe's own live boundary-cell pressure/temperature, and drives both the pipe network and the cylinder together one shared CFL-limited timestep at a time (`step_pipe_cylinder`/`run_pipe_cylinder_to_time`) — the same architecture `network.rs`'s `Junction` uses (override the reconstruction neighbor and face flux at the coupled end), generalized via a new, cylinder-agnostic `PipeNetwork::advance_with_external_fluxes`/`ExternalPortFlux` so `network.rs` itself never has to learn what a valve or cylinder is. The mass-flow-rate-to-`Flux` conversion (not needed before, since the old path only ever read a fixed reservoir) is built from the same `gas.rs` machinery every other flux in the solver uses, carefully keeping the valve's restricted throat area (feeds the orifice equation) and the pipe's own full bore area (converts the resulting kg/s into a flux density) from ever being conflated — verified by a standing invariant (`flux.mass * pipe_face_area == ±ṁ`, checked in a dedicated test). The cylinder-side energy update reads its rate directly off the *same* `Flux` value applied to the pipe (not a re-derived enthalpy formula) — a design review found that the old fixed-reservoir path's enthalpy-only formula is missing a kinetic-energy term a momentum-consistent pipe flux necessarily carries, which would have silently broken conservation as flow rate grew; reading both sides off one shared flux value (mirroring how `Junction` itself guarantees conservation) sidesteps that instead of patching the formula. Re-deriving HLLC's actual wall-reflection behavior by hand during this same review also caught a subtler, second finding: a naive "zero-lift flux equals a `ClosedEnd` boundary's flux" identity only holds exactly when the pipe's boundary-cell velocity is zero — a moving interior reflecting off a closed valve carries a real dynamic-pressure term this construction's "local static pressure only" closure doesn't capture, demonstrated (not just asserted) by a dedicated test showing the two genuinely diverge for nonzero interior velocity. Validated: (1) combined pipe+cylinder mass and energy conservation measured **exactly `0.0` relative error** (floating-point precision) over a short choked-flow window on a rigid cylinder — isolating the valve/pipe exchange from unrelated confounds (a real moving piston's own compression/expansion work is a genuine external energy source and was correctly excluded from this check, not folded in); (2) a single first coupled step's mass flow rate matches the pre-existing fixed-reservoir path's own analytic `valve::mass_flow_rate` call to **~1e-9 relative error**; (3) softer sanity checks confirm the coupled path tracks the already-validated fixed-reservoir path within **~1%** over a window short enough that the pipe's boundary state hasn't yet meaningfully evolved, and a closed-valve coupled trajectory tracks pure motoring to **~0.03%**.
   - **N-way branch junctions — done.** `branch_junction.rs`'s `BranchJunction` generalizes `network::Junction` (exactly 2 pipes, matching area) to any number of pipes with any areas — a single intake/exhaust runner splitting into several, or several merging into one. The closure was independently re-derived from OpenWAM's own N-way branch boundary (`TCCRamificacion.cpp`, used in this repo's real 6-cylinder 6-2-1 exhaust header case) and Gordon Blair's textbook (Sec. 2.13, "Benson's constant pressure model"): every connected leg shares one instantaneous junction pressure, each leg's own Riemann invariant (from its own boundary-cell state) determines its velocity at that shared pressure, and the pressure itself is found by bisection enforcing exact mass conservation. **Two more elaborate closures were tried and independently disproved before landing on this one** — a real design-review finding, not hidden: (1) giving a "receiving" leg a different (supplier's) sound speed in its own characteristic equation is provably wrong (with every leg at the same reference temperature, it forces the receiving leg's computed flow to have the *same sign* as the supplier's, so mass can never balance except degenerately); (2) resolving receiving legs via isentropic acceleration from a mass-flux-weighted mixed *stagnation* state (mirroring Blair's own more accurate energy-mixing equation) is also wrong, for a different reason — it gives every receiving leg the same velocity regardless of its own area, so conservation would require the supplying leg's area to exactly equal the total receiving area, a coincidence rather than a general solution. Both were caught by direct numerical scans of the total-mass-flow function showing it never actually crossed zero (or only crossed by geometric accident) rather than by inspection alone. The working closure — every leg using only its *own* entropy, unconditionally — passed that same scan cleanly for every case tried. Validated: (1) exact structural mass conservation (relative error ~1e-13, floating-point precision, for a 4-leg case with differing areas and pressures); (2) a symmetric N-branch split (and the reverse, N branches merging into one trunk) matches a closed-form linear-acoustics junction-pressure prediction (derived directly from Riemann-invariant matching at the junction) to within **0.07-0.6%** depending on disturbance amplitude, with the discrepancy shrinking at almost exactly **0.5x per halving of the disturbance** — confirming genuine convergence to the linear prediction, not a coincidence at one amplitude; (3) a quantified (not just sanity-checked) regression against `Junction`'s own exact 2-pipe HLLC solve for the matched-area `N=2` case; (4) a measured (not assumed-zero) energy-conservation residual — **~0.14%** at 1% disturbance amplitude, **~15.8%** at a realistic exhaust-pulse amplitude (3:1 pressure ratio) — an honest, real limitation of the shared-entropy closure (the same reason Blair's own more-accurate model needs a second, separately-solved energy equation), not engineered away.
   - **Lossy (non-isentropic) 3-way branch junction, closing that energy gap — done.** `lossy_branch_junction.rs`'s `LossyBranchJunction`/`resolve_lossy_branch_junction` implements Gordon Blair's "complete" model (Sec. 2.14, the non-isentropic "Bingham-Blair" closure) for the one topology he worked out in closed form: exactly 3 pipes, with a real angle-dependent pressure-loss term (`CL(θ) = 1.6 - 1.6·(θ/167)` for θ<167°, else 0) and a genuinely separate, jointly-solved stagnation-enthalpy energy equation, rather than `BranchJunction`'s single shared-entropy assumption. Two structurally different sub-cases, dispatched dynamically each resolve based on which leg(s) currently supply: **Case (a), one supplier + two supplied legs**, matches all 4 of Blair's own published test cases (Tables 2.14.1-2.14.5) — velocities/mass flows agree to within a documented **5-25%**, traced down to specific, ruled-out causes rather than left as a mystery: the shared leg-construction/Riemann-invariant machinery is independently confirmed correct (the lossless model, fed the identical incident-leg construction, reproduces Blair's own "constant pressure theory" table to <0.2%); reference-pressure gauge invariance for the shared-entropy unknown is proven algebraically, not just assumed; the Newton-Raphson solver provably converges to a genuine, near-machine-precision root of the coded equations in every case; and plugging Blair's own published numbers directly into his own momentum equation (no code involved at all) fails to self-balance by a similar 10-31%, strong evidence the gap is inherent to checking a steep `Ps = P0·Xs^7` power law against a table published to only 4 significant figures, not a formula error on either side. **Case (b), two suppliers + one supplied leg** (e.g. two exhaust runners merging into a collector), has no published reference anywhere in the source text and is validated only for internal consistency — exact `Ps1=Ps2` (Blair's own stated "assume equal pressure at the two supplier faces" simplification, confirmed independent of the angle between the two suppliers), correct mass-flow-direction signs, and smooth behavior under a parameter sweep. Rerunning `branch_junction.rs`'s own "realistic exhaust pulse" scenario (3:1 pressure ratio) through this model drops the energy residual from **~15.8%** to **~3e-7**. Getting the Newton-Raphson solver reliably convergent surfaced real, generalizable numerical-methods lessons, not just this-module trivia: a naive damped step can silently accept a `NaN` trial value once a component goes non-physical (Rust's `NaN < x` is always `false`, so a "step_scale too small, accept anyway" fallback needs an explicit finiteness check); a hard positivity clamp creates an exactly-flat Jacobian column right at the clamp boundary (fixed with a smooth, `C¹`-continuous exponential floor instead); leaving pressures (~1e5 Pa) and a sound speed (~1e2 m/s) unnormalized in the same Jacobian produces real, observed ill-conditioning; and plain step-scale damping can get stuck oscillating around a nonzero residual for hundreds of iterations when the Newton *direction* itself is poor, fixed by switching to proper Levenberg-Marquardt on the normal equations (`JᵀJ`, which unlike damping the raw Jacobian directly, guarantees a descent direction exists for large enough damping). One genuine, still-open scope limitation, found and documented rather than patched over: forcing Case (b)'s `Ps1=Ps2` exactly while each supplier keeps an independent entropy reference leaves a small residual that scales with how different the two suppliers' conditions are (consistent with `Ps1=Ps2` being Blair's own stated approximation, not an exact law) — and the doubly-degenerate case of *both* Case (a) angles sitting at the zero-loss cutoff simultaneously (a scenario Blair's own book never exercises, and a physically unusual geometry) reliably finds a spurious near-zero-flow root instead of the intended answer.
3. **Multi-cylinder firing order** — N-way junctions (above) are the architecture; firing order itself sets the relative phase of each cylinder's pulses arriving at a shared plenum/collector, checked against the 6-cylinder S54B32 OpenWAM case — that phase relationship is the entire physics of intake plenum and header/collector design.
4. **Muffler/silencer elements** — a genuinely different category from a tapered pipe: expansion chambers (sudden area change — the case explicitly deferred in phase 1's taper work, see above), perforated-tube sections, and side-branch resonators (Helmholtz/quarter-wave), each needing its own loss-coefficient/perforate-impedance model. Bundle in a revisit of `BoundaryCondition::Reservoir`'s simplified open-end treatment (see its doc comment) for proper reflection-coefficient accuracy — matters for acoustic muffler tuning in a way it didn't for the runner-only validation so far.
5. **Performance metrics + operating-point sweeps** — none of the above studies are judged from raw pressure/velocity/temperature traces directly; they're judged by volumetric efficiency, torque/power vs. RPM, trapped mass, pumping losses, and exhaust backpressure, derived automatically rather than read off a chart by hand. Needs support for running a single geometry across many RPM/load points (a torque curve is many solver runs, not one) — shapes `case.rs`'s API and the browser-side WASM compute budget for an optimization loop.
6. Wire the real solver into the web UI — `services/web/src/pages/SmokeTestPage.jsx` and `WavePage.jsx` still call the old placeholder API and show a broken/stale demo until this lands (chart design for wave profiles over time/crank-angle is a meaningfully different UI problem than a single sine curve, and deserves its own pass).

## Future cloud deployment

The web container is a single static SPA with no backend — deployable to any static host (Azure Static Web Apps, Netlify, GitHub Pages, S3+CloudFront, etc.) in addition to the Docker/nginx path above. No server-side compute, no COOP/COEP headers required (the wasm build is single-threaded for now — see `rust/crates/byglab-wasm`).
